# AGENTS.md — Правила проекта

Этот репозиторий основан на RobustToolbox / Space Station 14.

Файл описывает основные правила работы с кодовой базой и является источником истины для AI-помощника при разработке.

При изучении кода записывай ключевые выводы в файлы в .Codex/memory/. Ты также можешь использовать эти файлы при каждой новой сессии.
---

## ОБЩИЕ ПРИНЦИПЫ

- Следуй существующей архитектуре проекта. Не изменяй архитектуру без серьёзной необходимости.
- По возможности расширяй существующие системы, а не создавай новые.
- Все изменения должны быть минимальными и локальными.
- Не вводи новые паттерны, если они не используются в проекте.

---

## АРХИТЕКТУРА (ECS)

Проект использует Entity-Component-System:

- Entity — контейнер компонентов
- Component — только данные, без логики
- System — вся игровая логика

---

## СЛОИ ПРОЕКТА

- Content.Server — серверная логика (авторитетная)
- Content.Shared — общие компоненты и сетевые контракты
- Content.Client — клиентская часть (UI, визуализация, ввод)

---

## ПРАВИЛА СЕРВЕРА И КЛИЕНТА

### Сервер:
- Полностью управляет состоянием игры
- Выполняет симуляции и проверки
- Создаёт и удаляет сущности
- Является единственным источником истины

### Клиент:
- Отвечает за UI и визуализацию
- Обрабатывает ввод игрока
- Никогда не принимает решений о геймплее

### Shared:
- Компоненты
- Сетевые сообщения
- Общие enum и константы

---

## СЕТЕВОЕ ВЗАИМОДЕЙСТВИЕ

- Использовать networked components только для синхронизации состояния
- Минимизировать объём передаваемых данных
- Не передавать сущности целиком
- Предпочитать событийную модель вместо постоянного опроса

---

## UI ПРАВИЛА

- Использовать Bound User Interface (BUI)
- UI не должен изменять игровое состояние напрямую
- Поток данных:
  - Клиент → Сервер: сообщения
  - Сервер → Клиент: состояние UI

- Логика UI должна быть отделена от игровой логики

---

## КОМПОНЕНТЫ

- Компоненты содержат только данные
- Не допускается логика внутри компонентов
- Компоненты должны быть маленькими и специализированными
- Предпочтительна композиция, а не наследование

---

## СИСТЕМЫ

- Системы содержат всю игровую логику
- По возможности системы должны быть без состояния
- Избегать жёсткой связности между системами
- Использовать событийную модель вместо Update-поллинга

---

## ЛОГИКА ГЕЙМПЛЕЯ

- Вся критическая логика должна выполняться на сервере
- Клиент не может быть доверенным источником данных
- Все действия игрока должны валидироваться на сервере

---

## ВИЗУАЛИЗАЦИЯ

- Изменение спрайтов происходит только на клиенте
- Использовать Appearance / Visualizer системы
- Сервер не должен напрямую изменять SpriteComponent
- Сервер только задаёт состояние, клиент отображает

---

## СТИЛЬ КОДА

- Следовать существующему стилю проекта
- Избегать лишней абстракции
- Приоритет — читаемость
- Изменения должны быть точечными

---

## КАСТОМНЫЙ КОД ПРОЕКТА

- Весь кастомный код должен находиться в директориях:
  - _RuMC14 (основная логика кастома)
  - _RuMC14.Client (клиентская часть кастома)
  - _RuMC14.Server (серверная часть кастома, если используется разделение)

- Не размещать кастомный код в официальных директориях Content.*

---

## АПСТРИМ И ПОДДЕРЖКА

- Минимизировать изменения в официальном коде проекта
- Любая логика должна быть реализована через расширения, компоненты или системы
- Изменения базового кода допускаются только при крайней необходимости
- Проект должен оставаться максимально совместимым с апстримом SS14

---

## КОММЕНТАРИИ В КОДЕ

- Все комментарии писать на русском языке
- Комментарии должны объяснять "зачем", а не "что делает код"
- Избегать очевидных комментариев

---

## ПОВЕДЕНЧЕСКАЯ МОДЕЛЬ РАЗРАБОТКИ

При реализации задач:

1. Сначала анализируй существующий код
2. Используй уже существующие системы
3. Расширяй, а не переписывай
4. Избегай глобальных изменений
5. Сохраняй серверную авторитетность
6. Клиент оставляй только для отображения

---

## ПРИНЦИПЫ ПРОЕКТА

Проект ориентирован на:

- серверную детерминированную симуляцию
- чистое разделение ECS
- модульность систем
- простоту поддержки
- предсказуемое сетевое поведение

---

## ЕСЛИ НЕ УВЕРЕН

Если неясно, как реализовать задачу:

- ищи аналог в существующем коде
- придерживайся текущих паттернов проекта
- выбирай самое простое корректное решение
- не вводи новую архитектуру без необходимости
